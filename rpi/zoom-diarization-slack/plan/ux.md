# UX Design: Zoom Diarization Slack Service

**Feature Slug**: zoom-diarization-slack
**Date**: 2026-04-03
**Scope**: Zoom録音（m4a + VTT）からの話者分離付き議事録自動生成 → Slack投稿

> 本サービスはバックエンドのみで構成され、Web UI は存在しない。
> ユーザーとの接点は (1) 初期設定ファイル、(2) Slackチャンネルへの自動投稿、(3) エラー通知の3点に限定される。

---

## 1. サービス概要

### 1.1 システム構成

```
[Zoom クラウドレコーディング]
  ├── meeting_YYYY-MM-DD.m4a  (音声ファイル)
  └── meeting_YYYY-MM-DD.vtt  (文字起こし)
         │
         ▼  (自動アップロード)
[Google Drive 監視フォルダ]
         │
         ▼  (ポーリング検知)
[Zoom Diarization Slack Service]
  ├── VTTパース → タイムスタンプ付きセグメント
  ├── DiariZen話者分離 → 話者セグメント
  ├── VTT x 話者突合 → 話者ラベル付きセグメント
  ├── Claude API → 構造化議事録生成
  └── Slack Web API → チャンネル投稿
         │
         ▼
[Slack チャンネル]
  └── 議事録メッセージ（Block Kit フォーマット）
```

### 1.2 ユーザーの関与

| 操作 | 頻度 | ユーザーの作業 |
|------|------|-------------|
| 初期設定 | 1回 | `config_slack.yaml` を編集してサービスを起動 |
| 会議録音 | 毎回 | Zoomで録音開始（クラウドレコーディング有効） |
| 議事録確認 | 毎回 | Slackチャンネルで自動投稿された議事録を読む |
| エラー対応 | まれ | Slackに通知されたエラーを確認し、対処する |

ユーザーは**会議の録音を開始する以外に何もする必要がない**。録音終了後、Google Driveへの自動アップロード → サービスによる検知 → 議事録生成 → Slack投稿まで全自動で完了する。

## 3. Slackメッセージデザイン（成功時）

### 3.1 議事録メッセージ構成

議事録は1つのSlackメッセージとして投稿する。Block Kit を使用してセクションを明確に区切る。

```
+------------------------------------------------------------+
|                                                            |
|  :memo: 会議議事録 -- 2026-04-03 14:30                       |
|                                                            |
+------------------------------------------------------------+
|                                                            |
|  *参加者*                                                   |
|  Speaker A, Speaker B, Speaker C                           |
|                                                            |
+--- divider ------------------------------------------------+
|                                                            |
|  *まとめ*                                                   |
|  本会議では来月のリリーススケジュールについて議論した。          |
|  主な論点はQA期間の確保と外部APIの移行タイミングであった。      |
|  最終的に4月15日リリースで合意した。                          |
|                                                            |
+--- divider ------------------------------------------------+
|                                                            |
|  *主な議論ポイント*                                         |
|  - リリース日程: 4月15日（水）で確定                         |
|  - QA期間: 4月7日-14日の1週間を確保                          |
|  - 外部API移行: v2への移行を4月中に完了                      |
|  - ドキュメント: APIリファレンスの更新が必要                   |
|                                                            |
+--- divider ------------------------------------------------+
|                                                            |
|  *アクションアイテム / 次のステップ*                          |
|  :white_check_mark: Speaker A: QAテスト計画を4/5までに作成    |
|  :white_check_mark: Speaker B: API v2移行PRを4/7までに提出   |
|  :white_check_mark: Speaker C: ドキュメント更新を4/10までに   |
|                                                            |
+--- divider ------------------------------------------------+
|                                                            |
|  :bar_chart: *話者統計*                                     |
|  Speaker A  ██████████  12:30  2,456字                     |
|  Speaker B  ██████░░░░   7:45  1,234字                     |
|  Speaker C  ████░░░░░░   5:15    890字                     |
|                                                            |
+--- context -------------------------------------------------+
|  :clock3: 処理完了: 2026-04-03 15:05 | 録音時間: 25:30       |
+------------------------------------------------------------+
```

### 3.2 セクション詳細

| セクション | Block Kit種別 | 必須/任意 | 最大文字数 | 備考 |
|-----------|-------------|----------|-----------|------|
| タイトル | `header` | 必須 | 150文字 | 日時ベースの会議名 |
| 参加者 | `section` (mrkdwn) | 必須 | 3000文字 | v1では Speaker A/B/C ラベル |
| divider | `divider` | -- | -- | セクション区切り |
| まとめ | `section` (mrkdwn) | 必須 | 3000文字 | Claude APIが生成 |
| 主な議論ポイント | `section` (mrkdwn) | 任意 | 3000文字 | Claude APIが生成 |
| アクションアイテム | `section` (mrkdwn) | 任意 | 3000文字 | Claude APIが生成 |
| 話者統計 | `section` (mrkdwn) | 任意 | 3000文字 | コードブロックで等幅表示 |
| フッター | `context` | 必須 | -- | 処理時刻・録音時間 |

### 3.3 メッセージサイズ制限への対応

Slack APIの制限:
- 1メッセージあたり最大50ブロック
- `section` ブロックのテキスト上限: 3,000文字
- メッセージ全体: 約40,000文字

**分割戦略:**
1. 議事録全体が40,000文字以内 → 1メッセージで投稿
2. 40,000文字を超える場合 → メインメッセージ（タイトル + まとめ + アクションアイテム）を親メッセージとし、議論ポイント・話者統計をスレッドリプライとして投稿
3. 個別セクションが3,000文字を超える場合 → 3,000文字で切り詰め、末尾に「(続きは添付ファイルを参照)」を付与

## 4. ステータスメッセージ（処理中）

### 4.1 処理開始通知

ファイルペアリングが完了し処理を開始する時点で、Slackチャンネルに処理中メッセージを投稿する。

```
+------------------------------------------------------------+
|  :hourglass_flowing_sand: 会議録音を処理中...                 |
|                                                            |
|  ファイル: meeting_2026-04-03_143000.m4a                    |
|  ステージ: 音声解析中                                        |
+------------------------------------------------------------+
```

### 4.2 ステージ遷移

処理中メッセージは `chat.update` APIで同一メッセージを更新する（新規メッセージを投稿しない）。

| ステージ | 表示テキスト |
|---------|------------|
| ファイル検知 | `:hourglass_flowing_sand: 会議録音を検知しました（VTTファイルを待機中...）` |
| ペアリング完了 | `:hourglass_flowing_sand: 会議録音を処理中... ステージ: VTT解析` |
| VTTパース | `:hourglass_flowing_sand: 会議録音を処理中... ステージ: VTT解析` |
| 話者分離 | `:hourglass_flowing_sand: 会議録音を処理中... ステージ: 話者分離（DiariZen）` |
| VTT x 話者突合 | `:hourglass_flowing_sand: 会議録音を処理中... ステージ: 話者突合` |
| 議事録生成 | `:hourglass_flowing_sand: 会議録音を処理中... ステージ: 議事録生成（Claude API）` |
| Slack投稿 | (ステータスメッセージ削除 → 議事録メッセージ投稿) |

### 4.3 ステータスメッセージの削除

議事録の投稿が成功した場合、処理中メッセージは `chat.delete` APIで削除する。これによりチャンネルにノイズが残らない。

削除に失敗した場合はログに記録するのみで処理は続行する（ステータスメッセージは非クリティカル）。

## 5. エラー表示パターン

### 5.1 エラー分類と表示

| 分類 | Slack表示 | 処理 | 例 |
|------|----------|------|-----|
| 致命的エラー | エラーメッセージ投稿 | 処理中断 | Claude APIエラー、音声ファイル破損 |
| 部分的障害 | 警告付き議事録投稿 | フォールバックで続行 | 話者分離失敗、VTT欠損 |
| 一時的エラー | ユーザーには非表示 | 自動リトライ | API rate limit、ネットワーク一時障害 |
| 設定エラー | 起動時ログ出力 | サービス起動失敗 | 無効なトークン、フォルダID未設定 |

### 5.2 致命的エラーメッセージ

```
+------------------------------------------------------------+
|  :x: 議事録生成エラー                                        |
|                                                            |
|  ファイル: meeting_2026-04-03_143000.m4a                    |
|  失敗ステージ: generation                                   |
|  エラー: Claude API request failed (500 Internal Server     |
|         Error) after 3 retries                             |
|                                                            |
+--- context -------------------------------------------------+
|  :clock3: 2026-04-03 15:10                                  |
+------------------------------------------------------------+
```

### 5.3 部分的障害メッセージ

話者分離に失敗した場合、議事録は単一話者（Speaker）として生成し、注意書きを付与する。

議事録メッセージの先頭に警告ブロックを追加:

```
+------------------------------------------------------------+
|  :warning: 話者分離に失敗したため、単一話者として処理しました。  |
+------------------------------------------------------------+
|                                                            |
|  :memo: 会議議事録 -- 2026-04-03 14:30                       |
|  (以下、通常の議事録フォーマット)                              |
|                                                            |
```

VTTファイルが見つからずタイムアウトした場合:

```
+------------------------------------------------------------+
|  :warning: VTTファイルが見つからなかったため、音声のみで        |
|  処理しました。文字起こし精度が低下している可能性があります。    |
+------------------------------------------------------------+
|                                                            |
|  :memo: 会議議事録 -- 2026-04-03 14:30                       |
|  (以下、通常の議事録フォーマット)                              |
|                                                            |
```

### 5.4 エラーシナリオ一覧

| # | シナリオ | 検知方法 | 対応 | ユーザーへの表示 |
|---|---------|---------|------|----------------|
| 1 | VTTファイル欠損 | m4a検知後のペアリングタイムアウト（5分） | Whisperフォールバック（v2）/ VTTなしで処理 | 警告付き議事録 |
| 2 | DiariZen話者分離失敗 | DiariZen例外 / 空の出力 | 単一話者「Speaker」で続行 | 警告付き議事録 |
| 3 | Claude APIエラー | API例外（4xx/5xx） | 指数バックオフで3回リトライ | エラーメッセージ（3回失敗後） |
| 4 | Slack投稿失敗 | slack_sdk例外 | 指数バックオフで3回リトライ | ログ出力のみ（投稿手段がないため） |
| 5 | Google Drive認証失敗 | 認証例外 | ポーリング停止 + ログ出力 | エラーメッセージ（最後の投稿として） |
| 6 | ファイルペアリングタイムアウト | 5分間VTTが到着しない | 利用可能なファイルのみで処理 | 警告付き議事録 |
| 7 | 音声ファイル破損 | FFmpeg / DiariZenエラー | 処理中断 | エラーメッセージ |
| 8 | 音声が短すぎる（10秒未満） | ファイルサイズ / 再生時間チェック | 処理スキップ | なし（ログのみ） |

### 5.5 リトライ戦略

```
リトライ対象API:
  - Claude API: 3回、指数バックオフ（1s, 2s, 4s）
  - Slack Web API: 3回、指数バックオフ（1s, 2s, 4s）
  - Google Drive API: ポーリング自体がリトライの役割を果たす

リトライ対象外:
  - VTTパースエラー（データの問題であり再試行しても回復しない）
  - DiariZenモデルロード失敗（環境の問題）
  - 設定エラー（修正が必要）
```

## 7. ファイル添付仕様

### 7.1 文字起こし全文の添付

議事録メッセージの投稿後、文字起こし全文をスレッドリプライとしてファイル添付する。

| 項目 | 仕様 |
|------|------|
| ファイル名 | `transcript_YYYY-MM-DD_HHMMSS.md` |
| フォーマット | Markdown（話者ラベル付きタイムスタンプ） |
| 投稿先 | 議事録メッセージのスレッド（thread_ts指定） |
| API | `files.uploadV2` （`files.getUploadURLExternal` + PUT） |
| 最大サイズ | Slack Free: 5GB/ワークスペース、ファイル単体制限なし |

### 7.2 文字起こしファイルの内容

```markdown
# 文字起こし
- 日時: 2026-04-03 14:30
- 参加者: Speaker A, Speaker B, Speaker C
- 録音時間: 25:30

### 00:00:00

**Speaker A:** 今日の議題について確認します。
**Speaker B:** はい、まず予算の件からお願いします。

### 00:03:00

**Speaker C:** 予算案について説明します。先月の実績は...
**Speaker A:** なるほど、前年比で増加していますね。
```

### 7.3 議事録全文の添付（メッセージ分割時）

メッセージサイズ制限により議事録を切り詰めた場合、議事録全文もMarkdownファイルとしてスレッドに添付する。

| 項目 | 仕様 |
|------|------|
| ファイル名 | `minutes_YYYY-MM-DD_HHMMSS.md` |
| フォーマット | Claude APIが生成したMarkdown（切り詰めなし） |
| 投稿条件 | メッセージ内の議事録テキストが切り詰められた場合のみ |

### 7.4 ファイル添付のフロー

```
議事録メッセージ投稿（chat.postMessage）
  |
  |  thread_ts を取得
  |
  v
文字起こしファイル添付（files.uploadV2 → thread_ts指定）
  |
  v
(切り詰め発生時のみ) 議事録全文ファイル添付（files.uploadV2 → thread_ts指定）
  |
  v
完了
```

## 8. エッジケース対応

### 8.1 ファイルペアリング

Zoomはm4aとVTTを非同期にGoogle Driveにアップロードする。サービスはファイル名のプレフィックス（日時部分）でペアリングを行う。

| ケース | 挙動 |
|--------|------|
| m4aが先に到着、VTTが30秒後に到着 | m4a検知後、VTTを最大5分間待機。VTT到着後に処理開始 |
| VTTが先に到着、m4aが後に到着 | m4a到着をトリガーとしてペアリング。VTT単体では処理しない |
| m4aのみ到着（VTTなし） | 5分間のタイムアウト後、m4aのみで処理（VTTなしフォールバック） |
| VTTのみ到着（m4aなし） | 処理しない（音声ファイルが必須）。ログに記録 |
| 同日に複数会議の録音 | ファイル名のタイムスタンプ部分で区別してペアリング |

### 8.2 重複処理の防止

`StateStore`（既存モジュール）を再利用し、処理済みファイルのIDを記録する。

| ケース | 挙動 |
|--------|------|
| サービス再起動後に既存ファイルを再検知 | StateStoreで処理済みと判定、スキップ |
| 同一ファイルがリネームされた | Google DriveのファイルIDで判定するため重複しない |
| 処理中にサービスが停止 | 次回起動時に未完了ファイルを再処理（StateStoreで処理中状態を管理） |

### 8.3 長時間会議

| ケース | 挙動 |
|--------|------|
| 2時間超の会議 | DiariZen処理時間が増加（~10-15分）。全体タイムアウト（60分）で保護 |
| 議事録が40,000文字を超える | メッセージ分割（セクション3.3の分割戦略を適用） |
| Claude APIのトークン上限（max_tokens 4096）に到達 | 議事録が途中で切れる可能性あり。ファイル添付で全文を補完 |

### 8.4 同時処理

| ケース | 挙動 |
|--------|------|
| 複数会議のファイルが同時にアップロード | 逐次処理（キュー方式）。GPU VRAMの競合を回避 |
| 処理中に新しいファイルが到着 | キューに追加し、現在の処理完了後に開始 |

### 8.5 Slack固有の制限

| 制限 | 値 | 対策 |
|------|-----|------|
| メッセージ文字数上限 | ~40,000文字 | セクション3.3の分割戦略 |
| ブロック数上限 | 50ブロック/メッセージ | 議事録は通常10ブロック以下。超過時はテキストを結合 |
| Rate limit (chat.postMessage) | Tier 2: ~20 req/min | 通常利用では到達しない。到達時は指数バックオフ |
| Slack Free: メッセージ履歴 | 90日 | 制限事項として受け入れ（議事録の永続保存はスコープ外） |
| ファイルアップロードAPI | `files.getUploadURLExternal` + PUT | slack-sdk v3の推奨API（旧`files.upload`は非推奨） |

### 8.6 Zoom VTTの品質問題

リサーチレポートで指摘されたZoom VTTの日本語精度（20-40%エラー率の可能性）への対応:

| 品質レベル | 判定基準 | 対応 |
|-----------|---------|------|
| 良好（エラー率10%以下） | VTTセグメント数が十分、文字化けなし | 通常処理 |
| 許容可能（エラー率10-30%） | 部分的な誤認識あり | Claude APIが文脈補完。品質注記なし |
| 低品質（エラー率30%超） | 大量の誤認識、意味不明な文 | v2でWhisperフォールバック実装。v1では品質劣化を受け入れ |

v1ではVTT品質の自動判定は行わない。Claude APIの生成能力に委ね、低品質なVTTからでも要点を抽出する。

## 9. アクセシビリティ

### 9.1 テキストベースの出力

本サービスの出力は全てテキストベースであり、画像や動画を含まない。Slackの組み込みアクセシビリティ機能（スクリーンリーダー対応、キーボードナビゲーション、ハイコントラストテーマ）がそのまま適用される。

### 9.2 スクリーンリーダー互換性

- Block Kitの `header` ブロックはスクリーンリーダーで見出しとして認識される
- `section` ブロックの `mrkdwn` テキストはプレーンテキストとして読み上げ可能
- `context` ブロックは補足情報として適切にマークアップされる
- 話者統計のバーグラフ（`███░░░`）は数値（発言時間・文字数）を併記しており、視覚的表現のみに依存しない

### 9.3 キーボード操作

- Slackアプリ内でのメッセージ間移動、スレッド展開、ファイルダウンロードは全てキーボードで操作可能（Slackの標準機能）
- 本サービスにインタラクティブ要素（ボタン、セレクトメニュー等）はないため、追加のキーボード対応は不要

### 9.4 コントラストと可読性

- Slackのテーマ設定（ライト/ダーク）に自動対応（Block Kitはテーマに応じてレンダリングされる）
- `mrkdwn` の太字（`*テキスト*`）でセクション見出しを視覚的に区別
- コードブロック（` ``` `）で話者統計を等幅フォント表示し、数値の位置揃えを保証

### 9.5 言語

- 全てのメッセージテキスト、セクション見出し、エラーメッセージは日本語
- 話者ラベル（Speaker A, Speaker B）は英語だが、これは技術的な識別子であり、ローカライズの対象外
- Claude APIの議事録生成プロンプトは日本語であり、生成される議事録も日本語

## 10. 設定ファイル仕様

### 10.1 config_slack.yaml の構成

```yaml
# Zoom Diarization Slack Service 設定
slack:
  # Slack Bot Token (xoxb-...)
  bot_token: "xoxb-your-token-here"
  # 議事録投稿先チャンネルID
  channel_id: "C0123456789"

google_drive:
  # Service Account認証情報ファイルパス
  credentials_path: "credentials.json"
  # 監視対象フォルダID
  folder_id: "your_drive_folder_id"
  # ポーリング間隔（秒）
  poll_interval_sec: 30

diarization:
  # DiariZenモデル
  model: "BUT-FIT/diarizen-wavlm-large-s80-md"
  # 推論デバイス
  device: "cuda"

generator:
  # Claude APIモデル
  model: "claude-sonnet-4-5-20250929"
  # 最大トークン数
  max_tokens: 4096
  # Temperature
  temperature: 0.3

# ファイルペアリング
file_pairing:
  # VTTファイル待機タイムアウト（秒）
  vtt_timeout_sec: 300
  # ファイル名マッチングパターン
  m4a_pattern: "*.m4a"
  vtt_pattern: "*.vtt"

# 処理タイムアウト（秒）
processing_timeout_sec: 3600
```

### 10.2 起動時のバリデーションメッセージ

| 条件 | エラーメッセージ | 重大度 |

## 11. 既知の制限事項（v1）

### 11.1 匿名話者ラベル

v1では話者は Speaker A, Speaker B, Speaker C の匿名ラベルで表示される。声紋登録による実名マッピングはv2で実装予定。

**影響**: 議事録の可読性は実名表示と比較して劣る。ただし、Claude APIの議事録生成は発言内容から文脈を推定するため、まとめ/アクションアイテムの品質への影響は限定的。

### 11.2 Zoom VTTの日本語品質

Zoom VTTの日本語認識精度はZoom側の設定・音声品質に依存する。低品質な場合、議事録の精度も低下する。v2でWhisperフォールバックを検討。

### 11.3 Slack Free プランの制限

Slack Free プランでは90日でメッセージ履歴がアーカイブされる。議事録の永続保存が必要な場合はPro以上のプランが推奨される。

### 11.4 単一チャンネル対応

v1では1つのSlackチャンネルに対して1つのGoogle Driveフォルダを監視する構成。複数チャンネル/フォルダの対応はv2で検討。
