# Product Requirements: External Export (Google Docs)

**Feature Slug**: external-export
**Date**: 2026-03-17
**Status**: Planning
**Traceability**: R-82

---

## 1. Problem Statement

Meeting minutes generated by the bot are currently only accessible within Discord (as Embed + `.md` file attachment). Teams that use Google Workspace alongside Discord cannot easily share minutes with non-Discord stakeholders, archive them in their document management system, or collaborate on follow-up actions in Google Docs. Users must manually copy/paste or upload the `.md` file, which defeats the purpose of automated minute generation.

## 2. Goal

Automatically export generated meeting minutes to Google Drive as a Google Docs document immediately after Discord posting, so that team members can access, share, and collaborate on minutes through Google Workspace without any manual steps.

## 3. User Stories

### US-1: Automatic Google Docs Export (Must-Have)

**As a** meeting organizer,
**I want** meeting minutes to be automatically saved as a Google Docs document in a designated Drive folder,
**So that** non-Discord team members can access and reference the minutes.

**Acceptance Criteria**:
- After minutes are posted to Discord, a Google Docs document is created in the configured folder
- The document preserves headings, bullet lists, bold text, and checkboxes from the Markdown source
- The document title includes the meeting date (e.g., "Meeting Minutes -- 2026-03-17 14:00")
- The operation completes within 30 seconds of Discord posting

### US-2: Feature Toggle via Configuration (Must-Have)

**As a** bot administrator,
**I want** to enable or disable Google Docs export via `config.yaml`,
**So that** I can deploy the bot without Google API credentials if export is not needed.

**Acceptance Criteria**:
- `export.google_docs.enabled: false` is the default (opt-in feature)
- When disabled, no Google API calls are made and no errors are logged
- Enabling requires `folder_id` and `credentials_path` to be configured
- Invalid configuration triggers a clear validation error at startup

### US-3: Graceful Degradation on Export Failure (Must-Have)

**As a** bot administrator,
**I want** export failures to never block or disrupt Discord minutes posting,
**So that** the core pipeline remains reliable regardless of Google API availability.

**Acceptance Criteria**:
- Export runs after Discord posting completes (fire-and-forget, outside SLA)
- On failure, a warning is logged with error details
- Retries up to 3 times with exponential backoff
- Discord Embed and file attachment are unaffected by export failure

### US-4: Export Link in Discord Embed (Nice-to-Have, Phase 1 Stretch)

**As a** meeting participant,
**I want** the Discord Embed to include a link to the exported Google Docs document,
**So that** I can quickly navigate to the full document in Google Workspace.

**Acceptance Criteria**:
- If export succeeds, the Discord message is edited to add a "Google Docs" link field
- If export fails or is disabled, no link field appears
- Link editing failure is non-critical (logged as warning)

## 4. Scope

### In Scope (Phase 1)

| Item | Priority |
|------|----------|
| Google Drive HTML upload with automatic Docs conversion | Must-Have |
| Markdown-to-HTML conversion using `markdown` library | Must-Have |
| `ExportConfig` dataclass and `config.yaml` section | Must-Have |
| `ExportError` exception class | Must-Have |
| Pipeline integration (after archive, fault-tolerant) | Must-Have |
| Up to 3 retries with exponential backoff | Must-Have |
| Unit tests with mocked Google API | Must-Have |
| Export link appended to Discord Embed (edit-after-post) | Stretch |

### Out of Scope (Phase 2+)

| Item | Reason |
|------|--------|
| Notion export | Requires separate dependency and auth system; defer until demand validates |
| `/minutes export <message_id>` command | Manual re-export; add after core auto-export works |
| Per-guild export folder configuration | Multi-guild Drive export requires significant config complexity |
| Batch export of archived minutes | Depends on archive schema extension; separate feature |
| Google Docs API `batchUpdate` rich formatting | HTML auto-conversion is sufficient; only pursue if PoC shows quality issues |

## 5. Success Metrics

| Metric | Target | Measurement |
|--------|--------|-------------|
| Export success rate | > 95% | Logged export results over 30 days |
| Export latency | < 15s per document | Logged elapsed time |
| Pipeline impact | 0 additional failures | Compare pipeline success rate before/after |
| Adoption | Enabled in at least 1 guild config | Config audit |

## 6. Dependencies

| Dependency | Type | Status | Action Required |
|-----------|------|--------|-----------------|
| `markdown` Python library | New external | Not installed | `pip install markdown`, add to requirements.txt |
| `google-api-python-client` | Existing | Installed | None |
| `google-auth` | Existing | Installed | None |
| Google Drive API (write) | GCP Console | Read-only scope | Add `drive.file` scope to Service Account |
| Target Drive folder | Google Drive | N/A | Create folder, share with Service Account |

## 7. Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| HTML-to-Docs conversion quality | Medium | PoC validation required (Condition C1 from research) |
| Service Account scope upgrade | Low | `drive.file` only grants access to app-created files |
| `markdown` library dependency | Very Low | Mature, pure Python, 14+ year history |
| Google API rate limits | Very Low | Meeting frequency (a few per day) is far below limits |

## 8. Open Questions

1. **PoC validation**: Does Google Drive's HTML-to-Docs auto-conversion handle Japanese headings, `- [ ]` checkboxes, and `**bold**` text correctly? (Must verify before implementation)
2. **Scope strategy**: Should `drive.file` be a new scope alongside existing `drive.readonly`, or should `drive.readonly` be upgraded to `drive`? (Recommendation: add `drive.file` separately to maintain least-privilege)
3. **Embed editing**: Should the export URL be added via message edit, or should export run before posting so the URL is included in the initial Embed? (Recommendation: edit-after-post to avoid blocking the Discord post)
